## 二、核心设计：代码驱动规则 + LLM 驱动叙事

本项目的关键架构决策是将**确定性规则**和**创意叙事**分离到两个独立层：

| 职责 | 负责模块 | 原因 |
|------|----------|------|
| 数值变更边界（clamp） | GameEngine + Tool Executor | LLM 不可信任来做精确数值计算 |
| 判决走向评估 | GameEngine.evaluateVerdictOutlook | 确保可预测、可复现的游戏逻辑 |
| 重生触发 | GameEngine.maybeTriggerRebirth | 防止 LLM 任意触发/阻止重生 |
| 游戏结束判定 | GameEngine.checkGameOver | 50 轮硬上限 + 提前胜利条件 |
| 里程碑事件 | GameEngine.applyMilestones | 保证关键剧情节点按时发生 |
| 叙事文本生成 | AgentService → LLM | 需要创造力和上下文理解 |
| 选项生成 | AgentService → LLM (generate_choices) | 需要贴合当前叙事情境 |
| NPC 信任变化幅度 | AgentService → LLM (shift_npc_relation) | LLM 判断叙事情境，代码 clamp 幅度 |

## 三、Agent 设计：Tool Calling 循环

### 3.1 单轮流水线

```
玩家操作 (choiceId / userInput)
    │
    ▼
GameService.processTurn()
    ├── turn += 1, 计算 progress
    ├── writeMemoryFile() → data/{sessionId}.md
    │
    ├── AgentService.processTurn()
    │   ├── 构建 system prompt + user context JSON
    │   ├── Tool-calling loop (最多 3 轮):
    │   │   ├── LLM 返回 tool_calls[]
    │   │   ├── 逐个执行 → Tool Executor → 修改 GameState
    │   │   ├── 工具结果推回 messages[]
    │   │   └── 再次调用 LLM (带工具结果)
    │   ├── 解析最终 JSON: { narrative, summary }
    │   └── 校验必调工具 (update_stats, generate_choices)
    │
    ├── 后处理 (代码强制规则):
    │   ├── applyMilestones()      → 自动证据/事件
    │   ├── evaluateVerdictOutlook() → 判决走向
    │   ├── maybeTriggerRebirth()    → 重生检测
    │   ├── checkGameOver()          → 结局检测
    │   └── updateMemoryBundles()    → 记忆更新
    │
    ├── 记录 replay entry (含 token 统计)
    └── 持久化 → Repository
```

### 3.2 六个工具

| 工具 | 必选 | 输入 | 输出 | 状态变更 |
|------|------|------|------|----------|
| resolve_player_input | 仅自然语言输入时 | rawInput | resolvedChoiceId, interpretation, confidence | 无（仅解析） |
| update_stats | **每轮必调** | 五维 delta + fateDelta + reason | applied deltas | stats, rebirth.fate 修改 |
| generate_choices | **每轮必调** | choices[] (3-5个) | stored count | currentChoices 替换 |
| update_evidence | 可选 | action(add/update), evidence 字段 | 操作结果 | evidencePool 变更 |
| shift_npc_relation | 可选 | npcId, trustDelta, reason | trustAfter, stance | npcRelations 变更 |
| write_memory_anchor | 可选 | text (≤60 字) | stored, total | memory.longAnchors 追加 |

**必调工具校验**：如果 LLM 一轮内未调用 update_stats 或 generate_choices，抛出 `LLM_MISSING_TOOL` 错误并自动重试 1 次。

### 3.3 Delta 限幅（Tool Executor 强制）

```
truthScore, judgeTrust, publicPressure, evidenceIntegrity: ±10
juryBias: ±15
fate: ±8
NPC trust: ±15
```

所有 delta 在 Tool Executor 中 clamp 后再应用到 GameState，LLM 无法突破边界。

## 四、上下文管理策略（50 轮稳定的关键）

### 4.1 核心挑战

50 轮对话意味着上下文不断膨胀。如果每轮把完整历史传给 LLM，会导致：
- Token 消耗爆炸
- 超出模型上下文窗口
- LLM 在过长上下文中"迷失"，产生鬼打墙

### 4.2 解决方案：三层记忆 + 单轮无历史

**关键设计**：每次 LLM 调用**不传递对话历史**，仅传一条 system prompt + 一条 user message。所有历史信息压缩后嵌入 user message 的结构化 JSON 中。

```
messages = [
  { role: "system", content: AGENT_SYSTEM_PROMPT },     // 固定，可被缓存
  { role: "user",   content: JSON.stringify(context) }   // 每轮构建
]
```

user message 中的 context 包含：

| 字段 | 内容 | 长度控制 |
|------|------|----------|
| turn, progress | 当前轮次和章节 | 固定长度 |
| player | 玩家角色信息 | 固定长度 |
| playerAction | 本轮操作 | 短 |
| stats, flags | 当前数值和标记 | 固定长度 |
| evidencePool | 证据列表（优先级排序） | 最多 4 条 |
| npcRelations | NPC 关系 | 4 个 NPC |
| rebirth | 重生状态 | 固定长度 |
| recentHistory | 最近 2 轮摘要 | 每条 ≤90 字 |
| memoryContext | Markdown 记忆文件 | 截断至 1400 字符 |
| lastChoices | 上轮选项（防重复） | 最多 4 条 |
| worldContext | 世界观设定 | 固定（~800 字） |
| chapterScript | 当前章节剧本 | 仅当前层 + 前一层摘要 |

### 4.3 三层记忆系统

```
┌─────────────────────────────────────────────┐
│  短期记忆 (shortWindow)                      │
│  最近 6 轮的完整叙事文本                      │
│  用途：保持短期叙事连贯性                      │
│  淘汰：FIFO，超过 6 条自动移除最早的           │
├─────────────────────────────────────────────┤
│  中期摘要 (midSummary)                       │
│  每 5 轮生成 1 条摘要（来自 LLM summary）     │
│  最多 10 条                                  │
│  用途：保持剧情弧线和关键转折的中期记忆         │
│  淘汰：FIFO，超过 10 条自动移除最早的          │
├─────────────────────────────────────────────┤
│  长期锚点 (longAnchors)                      │
│  最多 8 条，由 LLM write_memory_anchor 或    │
│  代码里程碑事件自动写入                        │
│  用途：整局游戏的关键转折点永久记忆             │
│  淘汰：FIFO 或重生时按 memoryRetention 截断   │
└─────────────────────────────────────────────┘
```

### 4.4 Memory File（LLM 上下文注入）

每轮开始前，`writeMemoryFile()` 将三层记忆写入 `data/{sessionId}.md`：

```markdown
# 短期叙事
[最近 6 轮叙事]

# 中期摘要
[每 5 轮一条，最多 10 条]

# 长期锚点
[关键转折事件，最多 8 条]
```

读取时截断至 1400 字符（头部 70% + 尾部 30%），确保不会撑爆 user message。

### 4.5 重生时的记忆衰减

每次重生：
- `memoryRetention -= 6%`（下限 30%）
- `knownTruths` 按保留率截断（至少保留 1 条）
- 短期/中期记忆重置
- 长期锚点保留（通过 knownTruths 间接影响）

这使得每次重生既保留核心线索，又制造记忆模糊的不确定性。

### 4.6 章节真相注入优化

为避免后期 token 膨胀，章节剧本采用**滑动窗口**策略：
- 第 1-2 章：仅注入当前层真相
- 第 3+ 章：注入当前层 + 前一层首行摘要（非完整文本）

## 五、错误处理与降级策略

| 错误场景 | 处理方式 |
|----------|----------|
| LLM 未返回叙事 (LLM_NO_NARRATIVE) | 自动重试 1 次 |
| LLM 叙事格式无效 (LLM_INVALID_NARRATIVE) | 尝试从文本中提取 JSON，失败则重试 |
| LLM 未调用必选工具 (LLM_MISSING_TOOL) | 自动重试 1 次 |
| LLM API 调用失败 (LLM_ERROR) | 直接抛错，前端显示提示 |
| LLM 返回空响应 (LLM_EMPTY_RESPONSE) | 直接抛错 |
| 工具参数解析失败 | Tool Executor 返回错误结果，不崩溃 |
| 会话不存在 (SESSION_NOT_FOUND) | 404 错误 |
| 游戏已结束 (GAME_OVER) | 阻止继续操作 |

## 六、SSE 流式通信协议

前端通过 POST 请求触发回合，后端以 SSE 事件流返回：

```
event: input_feedback   → 玩家输入识别结果（置信度、解析结果）
event: narrative_delta   → 叙事文本分块（前端打字机效果 16ms/字符）
event: choices          → 下一轮选项列表
event: state_patch      → 完整游戏状态更新（数值、标记、证据、NPC、重生）
event: progress         → 章节进度更新
event: token_usage      → 本轮 Token 消耗（新增输入/缓存输入/输出）
event: done             → 回合结束信号 + 叙事摘要
event: game_over        → 游戏结束（结局类型 + 结局叙事）
```

## 七、Token 统计方案

### 7.1 三维度统计

| 维度 | 含义 | 来源 |
|------|------|------|
| inputTokens | 新增输入 Token（非缓存命中） | `usage.prompt_tokens - usage.prompt_tokens_details.cached_tokens` |
| cachedInputTokens | 缓存命中的输入 Token | `usage.prompt_tokens_details.cached_tokens` |
| outputTokens | LLM 输出 Token | `usage.completion_tokens` |

### 7.2 累加逻辑

由于每轮可能有多轮 tool-calling 循环（最多 3 轮），token 在循环中累加：

```typescript
for (let round = 0; round < MAX_TOOL_ROUNDS; round++) {
  const completion = await llm.call(...);
  totalPromptTokens += promptTokens - cachedTokens;
  totalCachedTokens += cachedTokens;
  totalCompletionTokens += completionTokens;
}
```

### 7.3 回放日志汇总

回放接口 `GET /sessions/:id/replay` 返回：
- 逐轮明细：每轮的 inputTokens / cachedInputTokens / outputTokens
- 全局汇总：总 Action 次数、三维度总计、每 Action 平均消耗

## 八、存储方案

### 8.1 文件存储（默认）

```
data/
├── sessions/           # {sessionId}.json → 完整 GameState
├── saves/              # {saveId}.json → SaveSnapshot（state 快照 + 时间戳）
└── {sessionId}.md      # Memory Markdown（LLM 上下文注入）
```

### 8.2 MongoDB（可选）

通过 `MONGODB_URI` 环境变量启用：
- `sessions` 集合：GameState 文档
- `saves` 集合：SaveSnapshot 文档
- 连接失败自动降级为文件存储
