## 三、这周遇到了些什么问题？

### 1. LLM 输出格式不稳定

最初设计 LLM 在最终回复中输出 JSON `{narrative, summary}`，但千问模型频繁出现格式错误（不合法 JSON、多余 Markdown 包裹等），导致 `parseNarrativeResponse` 要做大量容错。最终改为叙事纯文本直出 + `submit_summary` 工具调用，彻底绕开了 JSON 格式问题。

### 2. LLM 在叙事中夹带元信息

模型经常在叙事文本里列出选项清单、数值变化、摘要等结构化信息，破坏沉浸感。为此在 system prompt 中加了大量"严禁"规则（占了 prompt 相当篇幅），目前有改善但仍需持续调优。

### 3. 叙事篇幅控制

模型倾向于写长文，普通回合经常输出 400+ 字，节奏拖沓。加入了字数分级指导（常规轮 120~180 字、关键轮 200~280 字、终局轮最多 350 字），效果改善但还不完全稳定。

### 4. 流式输出中 tool_calls 增量拼接

千问的 streaming 模式下，tool_calls 以增量 delta 形式返回（分多个 chunk），需要按 index 累积拼接。这个实现比较 tricky，特别是同一轮可能同时产出 narrative content 和 tool_calls。

### 5. 存档系统设计

当前 `createSave` 只做快照不做标注，对玩家来说存档列表难以区分。正在改造中（加 label、listSaves、deleteSave），还没完成。

## 四、新的发现

1. **"工具调用 > JSON 输出"是 Agentic 游戏的关键设计模式**：让 LLM 把结构化信息（数值、选项、摘要）通过 tool calling 传递，叙事文本只承担纯文学创作，比让 LLM 输出混合 JSON 要稳定得多。这也让后端能对每类信息独立校验和重试。

2. **真相渐进解锁对叙事质量影响很大**：`chapterScripts.ts` 按章节只向 LLM 释放当前层真相，模型在第 1 章真的会写出"隐约察觉异常但不确定"的效果，而不是一上来就知道全部真相。这个设计在实际运行中效果比预期好。

3. **千问 qwen3.5-plus 的 tool calling + thinking 模式在中文法律叙事场景下表现不错**，能够在工具调用中做出合理的数值判断（比如"法官微微点头"对应 judgeTrust +3），但需要 system prompt 给出足够的示范和约束。

## 六、架构建议方向符合度分析

针对以下建议方向进行对照评估：
- 架构模块：主叙事Agent、数值管理模块、上下文管理模块
- 增强Tool：记忆管理、数值计算维护、回合/时间/进度管理、上下文压缩

### 1. 架构三模块对照

| 建议模块 | 当前实现 | 符合度 | 说明 |
|----------|----------|--------|------|
| **主叙事Agent** | `agentService.ts` | **高** | 已实现完整的 LLM tool-calling 循环（最多3轮），流式输出叙事 + 7个工具调用。LLM 负责叙事创作和选项生成，`agentSystemPrompt.ts` 有详细的角色、风格、字数约束。 |
| **数值管理模块** | `executor.ts` + `gameEngine.ts` | **高** | 数值变更全部通过 `update_stats`、`shift_npc_relation`、`update_evidence` 等工具执行，delta 双重 clamp（工具范围 + 全局范围）。NPC 信任自动推算立场。里程碑事件由确定性代码触发。LLM 不能直接改状态。 |
| **上下文管理模块** | `gameService.ts` 的 memory 逻辑 + `agentService.ts` 的 context 构建 | **中高** | 有三层记忆（short/mid/long），memory 文件截断策略（1400字符，头70%尾30%），证据池取 top 4，历史摘要取最近2条。但**没有独立模块**——逻辑分散在 gameService 和 agentService 中。 |

### 2. 建议增强Tool对照

| 建议Tool | 当前实现 | 符合度 | 差距 |
|----------|----------|--------|------|
| **记忆管理** | `write_memory_anchor` 工具 + 三层记忆体系 | **中高** | 已有长期锚点写入（max 8条）、中期摘要（每5轮）、短期窗口（最近6条）。**但缺少"召回"工具**——LLM 无法主动查询历史记忆，只能被动接收每轮注入的 `memoryContext`。 |
| **数值计算维护** | `update_stats` + executor clamp | **高** | 这正是当前架构的核心优势。LLM 通过工具提交 delta，代码负责 clamp 和校验，避免了"AI 口算"问题。判决走向、重生触发等关键数值完全由 `gameEngine.ts` 确定性公式计算。 |
| **回合/时间/进度管理** | `deriveProgress()` + 里程碑系统 | **中** | 有章节/场景推导（10场景×5章节）和7个硬编码里程碑。**但缺少独立的进度管理工具**——LLM 无法主动推进/查询进度，章节切换对 LLM 来说是被动感知的（通过 context 中的 progress 字段）。 |
| **上下文压缩** | memory 文件截断 + `submit_summary` 工具 | **中** | 有基础的压缩策略：摘要每轮生成、memory 文件截断、证据/历史限量。**但缺少主动压缩机制**——没有"当上下文过长时自动生成摘要替换原文"的能力，截断是硬切不是语义压缩。 |

### 3. 总结

**做得好的地方（与建议方向高度一致）：**
- "工具调用驱动状态变更" 是当前架构的核心设计，与"数值计算维护"建议完全吻合
- LLM 作为主叙事 Agent 的角色定位清晰，工具纪律由 system prompt + 必调校验双重保障
- 三层记忆体系（short/mid/long）概念完整

**可以进一步提升的方向：**

1. **上下文管理独立模块化**：当前 context 构建逻辑散在 agentService（构建 user message）和 gameService（memory 更新）中，可以抽成独立的 `ContextManager`，统一管理"给 LLM 看什么、裁多少、怎么排优先级"
2. **记忆召回工具**：增加一个 `recall_memory` 工具，让 LLM 能主动检索历史事件/证据关联，而不是只靠每轮被动注入的固定窗口
3. **语义压缩替代硬截断**：当前 1400 字符的 head/tail 截断会丢失中段信息，可以改为"当超长时用 LLM 生成压缩摘要"的方式
4. **进度管理增强**：目前章节推进是纯公式（turn/10），可以考虑加入基于剧情条件的章节切换（比如关键证据收集齐才进入下一章），让节奏更灵活

**整体评估**：当前实现已覆盖建议方向的 **70-80%**，核心的 Agentic 架构和数值管理做得扎实。主要差距在上下文管理的模块化程度和记忆系统的主动召回能力上。
