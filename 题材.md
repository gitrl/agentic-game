# 重生 + 法律悬疑 AI互动小说游戏设计方案

---

## 一、项目定位

本项目为一个基于 Agentic AI 的互动叙事系统，玩家通过选择或自然语言输入推动剧情发展。

核心特点：

* 第一人称沉浸叙事（"我"视角）
* 重生机制（多轮尝试 + 信息不完全保留）
* 法律悬疑推理（证据 + 证人 + 判决）
* 状态驱动剧情（非纯生成）

---

## 二、核心玩法循环

```
剧情生成（第一人称）
↓
提供选项 / 接收自然语言
↓
Rule Engine解析意图
↓
更新状态
↓
进入下一轮剧情
```

---

## 三、第一人称叙事设计

### 叙事要求

* 全程使用“我”视角
* 禁止使用“玩家”“你选择了”等表达

### 示例

```
我站在法庭中央，手心微微出汗。

这个案件，我经历过一次。

那一次，我判错了。

而这一次，我不能再犯错。
```

---

## 四、重生系统设计

### 状态结构

```json
{
  "loop": 2,
  "memory_retention": 0.6,
  "known_truths": ["嫌疑人A说谎"],
  "fate": 80
}
```

### 核心机制

1. **部分记忆保留**

   * 每次重生只保留关键线索

2. **命运阻力（fate）**

   * 改变越多 → 阻力越大

3. **重生触发条件**

   * 判决失败
   * 关键节点失败

## 七、Agent架构设计

### 1. Story Agent（叙事）

* 输出剧情（第一人称）
* 提供选项

### 2. Rule Engine（核心）

* 解析玩家输入
* 更新状态
* 控制数值变化

### 3. Memory Agent（记忆）

* 历史摘要
* 关键事件提取

### 4. Planner Agent（可选）

* 控制剧情节奏
* 插入反转

---

## 八、上下文管理策略

### 三层结构

1. 当前状态（结构化）
2. 历史摘要（压缩）
3. 最近对话（3轮）

### Prompt结构

```
【当前状态】
{state}

【剧情摘要】
{summary}

【最近对话】
{recent}
```

## 九、防止“鬼打墙”机制

1. 每轮必须推进事件：

   * 新证据
   * 新NPC
   * 新冲突

2. 剧情阶段控制：

```json
{
  "stage": "investigation → trial → verdict"
}
```

3. 限制重复：

   * 相同场景最多2轮
   * 相同选项不连续出现

## 十四、存档与恢复（页面刷新不丢进度）

### 目标

* 刷新页面后恢复到上一次轮次
* 上下文与状态一致（不影响后续生成）

### 存储策略（推荐）

**后端持久化 + 前端会话标识**

* 前端：保存 `session_id`（localStorage / cookie）
* 后端：以 `session_id` 为键存储完整游戏数据（DB/Redis）

### 存储内容（必须完整）

```json
{
  "session_id": "xxx",
  "round": 12,
  "state": { ... },
  "summary": "剧情摘要",
  "recent": [ ... 最近3轮对话 ... ],
  "log": [ ... 全量日志 ... ]
}
```

### 恢复流程

1. 页面加载 → 读取 `session_id`
2. 请求后端 `/load`
3. 恢复：

   * state（核心）
   * summary（中期记忆）
   * recent（短期记忆）
4. 重新渲染最后一轮剧情 + 选项

### 存档时机（关键）

每一轮玩家Action后：

```
玩家选择 → 状态更新 → AI生成 → 写日志 → 持久化
```

### 技术实现建议

* MVP：

  * localStorage（前端）+ JSON（后端文件）
* 进阶：

  * Redis（高性能）
  * 数据库（MySQL / MongoDB）

### 防坑点（必须注意）

* ❌ 不要只存对话文本（会崩）
* ✅ 必须存结构化 state + summary
* ❌ 不要重新让AI“回忆历史”
* ✅ 直接恢复状态再继续生成

### 额外加分点

* 多存档位（slot1 / slot2）
* 自动存档 + 手动存档
* 回放模式（基于日志）

本项目核心在于：

> 用结构化系统约束AI生成，使其在50轮内持续稳定地产生有逻辑、有推进、有沉浸感的互动叙事。
