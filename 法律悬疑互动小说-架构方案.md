# 法律悬疑互动小说游戏架构方案

## 1. 题材定稿

- 项目名建议：`逆证：最后五十轮`
- 题材定位：法律悬疑互动小说
- 玩家身份：辩护律师（兼调查者）
- 核心玩法：证据判断 + 质证对抗 + 法庭攻防
- 主流程：案发 -> 调查取证 -> 证人交锋 -> 法庭辩论 -> 判决
- 多结局：
  - 真相大白
  - 冤案
  - 被误导
  - 权力干预

## 2. 总体架构

### 2.1 前后端边界

- 前端（React）：
  - 仅负责渲染与交互
  - 展示叙事实时流、选项、进度、状态面板、回放
- 后端（Node.js + TypeScript）：
  - 负责会话管理、Agent编排、状态机、日志、Token统计、存档

### 2.2 核心系统组件

- API 服务层：处理会话与回合请求
- Agent 编排层：统一调度各个 Agent
- 规则与状态层：数值更新、结局判定、进度推进
- 存储层：会话状态、回放日志、存档快照、设定数据
- LLM Gateway：统一大模型调用与用量统计

## 3. Agent 设计

| Agent | 主要职责 | 关键输入 | 关键输出 | 异常处理 |
|---|---|---|---|---|
| Orchestrator Agent | 回合总控，编排调用顺序 | 会话状态、玩家动作 | 回合结果包 | 降级到安全模板流程 |
| Case Planner Agent | 规划本轮案件推进点 | 章节信息、证据链、NPC关系 | 结构化剧情骨架 | 重试后走锚点推进 |
| Evidence Analyst Agent | 分析证据可信度与冲突 | 证据池、证词、时间线 | 证据结论、风险标签 | 标记证据不足并建议补证 |
| Narrative Writer Agent | 生成叙事文本并流式输出 | 剧情骨架、状态快照 | narrative 分段文本 | 使用短模板兜底不断流 |
| Choice Designer Agent | 生成下一轮可选动作 | 当前局势、可操作资源 | 3-4个结构化选项 | 返回保底通用选项 |
| State Referee Agent | 裁定数值与事件变化 | 玩家选择、规则表 | state_patch、event_tags | 规则引擎兜底重算 |
| Consistency Auditor Agent | 检查矛盾、重复、越界 | 本轮草案、历史摘要、状态 | 审计结果、修复建议 | 触发 Recovery Agent |
| Recovery Agent | 修复异常输出 | 审计失败信息、原输出 | 修复后的叙事与选项 | 失败时回退安全回合 |
| Memory Curator Agent | 上下文压缩与锚点维护 | 多轮历史、关键事件 | short/mid/long 记忆包 | 硬锚点优先保留 |

## 4. 非 Agent 模块（规则化实现）

- Rule Engine：
  - 数值计算
  - 条件触发
  - 结局判定
- Progress Manager：
  - 章节与幕推进
  - 50轮节奏控制
- Replay Logger：
  - 每轮操作记录
  - 摘要和事件标记
  - 数值变化追踪
- Token Meter：
  - 单请求 token 明细
  - 单 Action 汇总
  - 全局统计输出
- Save Manager：
  - 存档快照写入
  - 读档恢复会话

## 5. 每轮执行流水线（Turn Pipeline）

1. 前端提交玩家 action
2. Orchestrator 读取会话状态与记忆包
3. Case Planner 产出本轮剧情骨架
4. Evidence Analyst 更新证据冲突关系
5. State Referee 生成状态变化草案
6. Narrative Writer 生成流式叙事文本
7. Choice Designer 生成下一轮选项
8. Consistency Auditor 审计一致性
9. 失败则调用 Recovery 修复
10. Rule Engine 最终校验并提交状态
11. Replay Logger + Token Meter 记录落盘
12. SSE 推送给前端：
    - `narrative_delta`
    - `choices`
    - `state_patch`
    - `progress`
    - `token_usage`
    - `done`

## 7. 法律题材核心数值

- `truth_score`：接近真相程度
- `judge_trust`：法官对辩方可信度
- `jury_bias`：陪审倾向（可正可负）
- `public_pressure`：舆情与外部压力
- `evidence_integrity`：证据链完整度

规则要求：

- 每个选项至少影响 1-2 个核心数值
- 每轮必须产生可观测推进（事件、关系、证据、位置至少一项变化）

## 8. 结局判定策略

- 真相大白：
  - `truth_score` 高
  - `evidence_integrity` 高
  - 程序污染标记低
- 冤案：
  - 真相线索不足或关键证据未建立
  - 最终判决有罪
- 被误导：
  - 伪证或错误动机被采纳
  - 核心推理线偏离
- 权力干预：
  - `public_pressure` 爆表或触发干预 flag
  - 程序正义受损

## 9. 50轮稳定策略

- 三层记忆：
  - 短期窗口：最近 4-6 轮原文
  - 中期摘要：每 5 轮压缩摘要
  - 长期锚点：关键事实、证据、关系、伏笔
- 反重复机制：
  - 检测叙事相似度与选项相似度
  - 超阈值强制引入新证据或新冲突
- 结构化输出约束：
  - Agent 输出遵循 JSON Schema
  - 解析失败自动重试与降级
- 先审计再落库：
  - 一致性不通过不写入主状态

## 10. 日志与评测对齐

每轮日志必须包含：

- 玩家操作（选项或输入）
- AI叙事摘要
- 数值变化
- 关键事件标签
- Token 明细：
  - 新增输入
  - 缓存输入
  - 输出

全局统计：

- 总 Action 次数
- 三维 token 总计
- 每 Action 平均消耗

该结构可直接支持评测维度：

- 稳定性
- 剧情推进率
- 鲁棒性
- Token效率

## 11. 设计阶段交付物（先不编码）

1. 世界设定包 v1（时代、案件、势力、NPC、章节锚点）
2. Agent I/O 文档（输入输出字段和错误码）
3. Prompt 文档（系统提示、风格约束、审计提示、修复提示）
4. 异常处理矩阵（无响应、格式错误、逻辑矛盾、鬼打墙）
5. 回放日志 JSON Schema 与评测导出格式
