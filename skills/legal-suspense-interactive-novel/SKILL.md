# 法律悬疑互动小说设计

输出可直接进入实现的设计产物，同时保持评审可读性与落地性。

## 工作流

1. 先将请求归类到以下三类之一。
- 从概念到架构：用户提供题材、创意或课程要求。
- 现有文档升级：用户提供已有方案并要求优化。
- 交付准备：用户要求检查清单、评测格式或里程碑计划。

2. 在写架构前先锁定硬约束。
- 确认目标稳定轮数（`>=30` 或 `>=50`）。
- 确认运行时组件（前端、后端、模型网关、持久化）。
- 确认必交付项（日志、回放、Token 统计、存读档）。

3. 先搭系统骨架，再补细节。
- 先划分模块边界，再分配职责。
- 为每个 Agent 定义结构化输入输出与降级策略。
- 固定一条标准回合流水线，并明确失败恢复路径。

4. 以结构优先方式定义数据与规则。
- 产出字段清晰、类型明确的状态模型，并注明更新责任方。
- 将数值系统与布尔标记系统分开定义。
- 用规则强制“每轮必须推进”，不要只写叙事描述。

5. 定义结局与评测口径。
- 用“阈值 + 标记”组合表达各结局条件。
- 增加反循环约束与一致性审计。
- 增加可观测性要求：回放记录、Token 明细、每轮摘要。

6. 输出面向实现的交付件。
- 世界观设定包大纲。
- Agent I/O 与错误码表。
- 提示词包（system/style/audit/recovery）。
- 异常处理矩阵。
- 回放与评测导出 Schema。

## 输出顺序

除非用户另有要求，按以下顺序输出：

1. 范围与假设。
2. 架构摘要。
3. 回合流水线。
4. 状态与规则。
5. 结局与防失败策略。
6. 交付清单。

## 质量门槛

在最终输出前执行以下检查：

- 每个 Agent 都有明确输入、输出、降级。
- 回合流水线包含解析失败与一致性失败路径。
- 每次玩家动作至少改变一个可量化维度。
- 各结局具备可区分性，不重叠。
- 回放与 Token 字段足以支持评测与排障。
